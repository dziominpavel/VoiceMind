## Context

- `Reminder` — одноразовое (`fireAt`, `status`), без правила повторения.
- `ReminderScheduler` ставит один alarm; после fire напоминание уходит в TRIGGERED/историю.
- `ReminderParser` извлекает одно время; повторения не распознаёт.
- Точность: `setExactAndAllowWhileIdle` + fallback; есть `BootReceiver` + `rescheduleAll`.

## Goals / Non-Goals

**Goals:**
- Поддержать частые правила повторения: ежедневно, по будням, по выходным, еженедельно по дню недели, ежемесячно по числу; с интервалом (каждые N).
- Распознавание этих правил голосом.
- Надёжный пересчёт следующего срабатывания и перепланирование.

**Non-Goals:**
- Полный RFC 5545 RRULE и сложные исключения (EXDATE) — за рамками MVP.
- Произвольные комбинации («каждый 2-й вторник месяца») — позже.
- Синхронизация с системным календарём.

## Decisions

### 1. Хранение правила

- Поле `recurrenceRule: String?` в `Reminder` (null = одноразовое). Формат — компактная собственная строка/enum-сериализация: тип (`DAILY|WEEKDAYS|WEEKENDS|WEEKLY|MONTHLY`) + параметры (день недели/число, интервал).
- **Альтернатива (отклонена для MVP):** полноценный RRULE — избыточно сложно; подключим позже без слома (поле остаётся строкой).

### 2. Вычисление следующего срабатывания

- Чистая функция `nextOccurrence(rule, fromMillis): Long` в локальной TZ через `Calendar`/`java.time`.
- Считать строго в будущее относительно `now` (а не `fireAt+период`), чтобы пропуски (Doze/выключенный телефон) не накапливали серию прошлых срабатываний; при пропуске — ближайшее будущее по правилу.

### 3. Жизненный цикл при срабатывании

- На fire повторяющегося: показать оповещение как обычно, затем НЕ уводить в историю, а вычислить `nextOccurrence` и перепланировать (`status = PENDING`, новый `fireAt`).
- Действие «остановить повторение» снимает `recurrenceRule` и переводит в обычный жизненный цикл.
- `BootReceiver.rescheduleAll` корректно планирует следующее срабатывание для повторяющихся.

### 4. Парсинг

- Расширить `ReminderParser`: триггеры «каждый/каждые/ежедневно/по будням/по выходным/каждый <день недели>/каждое <число>». Время берётся из существующего парсинга времени; при отсутствии времени — вести в подтверждение.
- Покрыть юнит-тестами в стиле `ReminderParserTest`.

## Risks / Trade-offs

- **[DST/часовые пояса]** → расчёт через календарь локальной TZ, не ручная арифметика; тесты на границах DST.
- **[Пропущенные срабатывания в Doze]** → `nextOccurrence` от `now` в будущее; опираемся на reliability-трек (battery exemption) для точности.
- **[Миграция Room]** → только добавление nullable-поля, не destructive; версия +1, тест миграции.
- **[Накопление «хвоста» прошлых]** → не планируем прошлые; всегда ближайшее будущее.

## Migration Plan

1. Room: добавить поле(я) повторения, поднять версию, написать миграцию (по `/db-migration`) + тест.
2. Реализовать `nextOccurrence` + юнит-тесты.
3. Встроить пересчёт в fire-путь и `rescheduleAll`.
4. Парсер: правила повторения + тесты.
5. UI: показ/снятие повторения, строки.
6. Сборка/тесты, ручная проверка ежедневного и «по будням».
- **Rollback:** поле nullable; отключение распознавания и пересчёта возвращает одноразовое поведение (данные совместимы).
