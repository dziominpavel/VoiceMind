## Context

Текущее состояние:
- `ReminderPermissions` умеет проверять/открывать exact alarm, notifications, full-screen intent, но **не** знает про оптимизацию батареи.
- `MainActivity` при старте тихо запрашивает только `POST_NOTIFICATIONS` (`LaunchedEffect(Unit)`).
- Экрана онбординга нет; `SettingsScreen` показывает статусы разрешений, но не ведёт пользователя проактивно.
- Манифест объявляет `SCHEDULE_EXACT_ALARM`, `USE_FULL_SCREEN_INTENT`, `POST_NOTIFICATIONS`, но не `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`.
- `ReminderScheduler` уже делает fallback на `setAndAllowWhileIdle` при отсутствии exact-alarm.

## Goals / Non-Goals

**Goals:**
- При первом запуске провести пользователя через критичные для срабатывания разрешения с объяснением «зачем».
- Дать возможность запросить исключение из оптимизации батареи и проверить его состояние.
- Дать «тестовое напоминание», подтверждающее реальную доставку.
- Показывать постоянный индикатор, если критичных разрешений не хватает.
- Сделать мастер повторно доступным из настроек.

**Non-Goals:**
- Per-OEM автоматизация autostart (только ссылки/подсказки, без хаков под конкретные прошивки) — можно отдельным инкрементом.
- Изменение модели доставки/каналов уведомлений.
- Foreground-service или иные тяжёлые механизмы удержания (за рамками; alarm + exact + battery exemption достаточно для MVP).

## Decisions

### 1. Запрос исключения из оптимизации батареи

- Проверка: `PowerManager.isIgnoringBatteryOptimizations(packageName)`.
- Запрос: `Intent(Settings.ACTION_REQUEST_IGNORE_BATTERY_OPTIMIZATIONS)` с `package:`-data — показывает системный диалог сразу. Требует разрешения `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` в манифесте.
- **Обоснование для Google Play:** core-функция приложения — будильник/напоминание; это попадает под разрешённое исключение политики IBO. Если посчитаем рискованным для публикации — fallback на `ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS` (открывает список, без спец-разрешения).
- **Альтернатива (отклонена для шага запроса):** только открывать общий экран настроек — хуже по конверсии, пользователь теряется.

### 2. Онбординг как оверлей, управляемый состоянием

- Состояние «онбординг пройден» хранится флагом в `SettingsRepository` (`onboarding_completed`).
- Мастер показывается, если флаг не выставлен ИЛИ при явном вызове из настроек.
- Реализация — Compose-оверлей (как существующие Confirm/Manual/Detail overlay), один источник состояния во `VoiceMindViewModel`. Бизнес-логику разрешений не держим в Composable: ViewModel отдаёт «какие шаги нужны», UI запускает системные launcher'ы.
- Шаги динамические: показываем только невыданные/неактуальные для текущего API (например, `POST_NOTIFICATIONS` только на 33+, exact alarm только на 31+).

### 3. Тестовое напоминание

- Кнопка ставит обычный `Reminder` с `fireAt = now + ~60s`, `body = "Проверка VoiceMind"`, через существующий `ReminderRepository.insertAndSchedule` (тот же путь, что реальные напоминания — честная проверка).
- Не вводим отдельный «тестовый» тип; используем обычный поток, чтобы проверка отражала реальную доставку. Пользователь может его удалить/оно само станет историей.

### 4. Индикатор надёжности на Home

- ViewModel выставляет производное состояние `reliabilityIssues` (нет уведомлений / нет exact alarm / батарея не исключена).
- `HomeScreen` показывает баннер с кнопкой «Исправить» → открывает мастер. Баннер скрывается, когда всё ок.
- Проверять состояние на `onResume` (разрешения могли измениться в системных настройках).

## Risks / Trade-offs

- **[Play отклонит `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS`]** → держим fallback на `ACTION_IGNORE_BATTERY_OPTIMIZATION_SETTINGS`; решение о публикации — за владельцем.
- **[Навязчивость онбординга]** → один раз при первом запуске, пропускаемый; повтор только по запросу из настроек.
- **[Состояние разрешений меняется вне приложения]** → перечитывать на `onResume`, не кэшировать жёстко.
- **[Тестовое напоминание засоряет историю]** → короткий понятный текст; пользователь может удалить; приемлемо.
- **[Фрагментация OEM autostart]** → вне scope автоматизации; даём текстовую подсказку и ссылку, не обещаем 100%.

## Migration Plan

1. Добавить permission в манифест + battery-хелперы в `ReminderPermissions`.
2. Добавить флаг `onboarding_completed` в `SettingsRepository`.
3. ViewModel: состояние шагов мастера + `reliabilityIssues` + событие показа.
4. UI: оверлей мастера, баннер на Home, точка входа из настроек, обработчики launcher'ов в `MainActivity`.
5. Тестовое напоминание через репозиторий.
6. Строки, ручная проверка на устройстве с включённой оптимизацией батареи.
- **Rollback:** функционал изолирован (новый оверлей + флаг + хелперы); отключается удалением точки показа мастера и баннера.
