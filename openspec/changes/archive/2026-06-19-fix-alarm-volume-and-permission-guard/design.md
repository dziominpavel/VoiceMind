## Context

Два независимых дефекта в потоке доставки и сохранения:

1. **Громкость.** `AlarmVolumeSlider` в `SettingsScreen.kt` объявлен с `valueRange = 0f..15f, steps = 14`, то есть отдаёт значения `0..15`. `SettingsRepository.setAlarmVolume` коэрцит в `0..100` и хранит как есть. `AlarmSoundPlayer.play(volumePercent)` считает `targetVolume = maxStreamVolume * volumePercent / 100`. Итог: касание слайдера ограничивает громкость значением `≤ 15%`. Дефолт `100` спасает только до первого взаимодействия.

2. **Тихое сохранение.** `VoiceMindViewModel.saveReminder` при отсутствии `POST_NOTIFICATIONS` пишет в `_errorMessage`, затем (через `else if`) при отсутствии точных будильников перетирает его другим текстом, после чего безусловно выполняет `safeDb { … insertAndSchedule }`. Сообщение транзиентное (Snackbar), часто перетирается и не действенно. Пользователь получает «успешно сохранено», хотя уведомление может не показаться вовсе (нет `POST_NOTIFICATIONS`).

Это pre-release MVP, поэтому допустимы простые одноразовые миграции данных без обратной совместимости со «старым» масштабом.

## Goals / Non-Goals

**Goals:**
- Громкость будильника управляется в процентах `0..100`; слайдер и хранимое значение в одном масштабе с `AlarmSoundPlayer`.
- Существующие значения `alarm_volume` (старый масштаб `0..15`) один раз корректно переводятся в проценты.
- Сохранение напоминания без `POST_NOTIFICATIONS` перестаёт быть «молча успешным»: пользователь получает действенное предупреждение и запрос разрешения; ввод не теряется.
- Отсутствие точных будильников не блокирует сохранение (остаётся fallback на неточный alarm), но факт неточности остаётся виден пользователю.

**Non-Goals:**
- Постоянный баннер надёжности на главном/в списке (вынесено в будущий change «reliability onboarding»).
- Изменение схемы Room, новые разрешения в манифесте.
- Per-OEM autostart-инструкции, исключение из оптимизации батареи (отдельный change).

## Decisions

### 1. Громкость — единый масштаб «проценты 0..100»

- `AlarmVolumeSlider`: `valueRange = 0f..100f`. Дискретность — шаги по 5% (`steps = 19`) для удобного попадания; либо непрерывный слайдер. Выбор: шаги по 5% (предсказуемо, без «дрожания» значения).
- Хранение и `AlarmSoundPlayer` уже работают в процентах — менять их трактовку не нужно, только привести слайдер к тому же масштабу.
- **Альтернатива (отклонена):** перевести всё в «системные шаги `STREAM_ALARM`» (`getStreamMaxVolume`). Минусы: max зависит от устройства (7/15/…), хуже для UI и для хранимого значения; проценты нагляднее и стабильнее.

### 2. Миграция старых значений громкости

- Одноразовая миграция в `SettingsRepository`: при чтении, если флаг `alarm_volume_migrated_v2` не выставлен и хранимое `alarm_volume` присутствует и `<= 15`, пересчитать `percent = round(value / 15 * 100)`, сохранить и выставить флаг.
- Значения, заданные уже в новом масштабе (`> 15`) или отсутствующие (дефолт 100), не трогаются.
- **Trade-off:** пользователь, осознанно выставивший очень тихую громкость в старом масштабе, после миграции получит её пересчитанной (например, было «10» → станет ~67%). Для pre-release MVP приемлемо; вероятность мала, последствие — будильник громче, что безопаснее для напоминалки.

### 3. Гард разрешений при сохранении

- Разнести проверки: отсутствие `POST_NOTIFICATIONS` и отсутствие точных будильников — независимые сигналы, нельзя перетирать один другим.
- **POST_NOTIFICATIONS отсутствует:** напоминание всё равно сохраняется (не терять ввод), но система MUST показать действенное предупреждение (Snackbar с действием «Разрешить»/переход в настройки) и инициировать запрос разрешения. Не помечать сохранение как тихий успех.
- **Точные будильники отсутствуют:** сохранять и планировать через fallback (`setAndAllowWhileIdle`, уже реализовано), но показать предупреждение о возможной неточности.
- Если отсутствуют оба — показать оба сигнала (объединённый текст), без потери ни одного.
- Технически: вынести результат проверки разрешений из `saveReminder` в состояние/событие ViewModel, чтобы UI мог запустить системный запрос разрешения (launcher уже есть в `MainActivity`), а не только показать текст.

## Risks / Trade-offs

- **[Миграция громкости перетрёт осознанно тихое значение]** → ограничиваем условием `<= 15` и одноразовым флагом; в худшем случае будильник станет громче (безопасное направление для напоминалки).
- **[Слайдер 0..100 непрерывный «дёргается»]** → используем шаги 5% (`steps = 19`).
- **[Запрос разрешения из потока сохранения усложняет ViewModel↔UI]** → используем уже существующий `notificationPermissionLauncher` в `MainActivity`; ViewModel лишь выставляет событие/флаг «нужно запросить разрешение».
- **[Регресс громкости на устройствах с малым maxStreamVolume]** → формула `maxVolume * percent / 100` уже корректна; меняем только источник `percent`.

## Migration Plan

1. Добавить ключ-флаг `alarm_volume_migrated_v2` в `SettingsRepository`; реализовать одноразовый пересчёт при чтении/инициализации.
2. Поправить диапазон слайдера на `0..100` (шаги 5%).
3. Переписать ветку разрешений в `saveReminder` (независимые сигналы + событие запроса разрешения, сохранение не «молчит»).
4. Прогнать `:app:testDebugUnitTest` и ручной чек: выставить громкость на максимум → звонок громкий; создать напоминание без разрешения на уведомления → видно действенное предупреждение и запрос.
- **Rollback:** изменения изолированы в трёх файлах + один ключ DataStore; откат — обратные правки, флаг миграции безвреден.
