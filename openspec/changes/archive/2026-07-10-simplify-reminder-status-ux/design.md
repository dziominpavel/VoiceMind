## Context

Сейчас в коде четыре статуса (`PENDING`, `TRIGGERED`, `DONE`, `CANCELLED`), но UI и specs говорят на смеси старых имён (`SCHEDULED`/`FIRED`/`DISMISSED`) и относительных подписей. В Истории `formatRelativeFireAt()` для любого `fireAt` в прошлом возвращает «просрочено» — поэтому почти все карточки выглядят одинаково «просроченными», хотя статусы разные. Кнопки «Готово» и «Отменить» в шторке семантически различны (DONE vs CANCELLED), но для пользователя в момент срабатывания выглядят как два способа «убрать уведомление».

## Goals / Non-Goals

**Goals:**
- Единая пользовательская модель статусов с понятными русскими лейблами.
- История показывает **статус + дату**, никогда не «просрочено».
- «Просрочено» только для живых `PENDING` в Предстоящих.
- Кнопка «Выполнено» вместо «Готово» везде (уведомление, alarm UI, детали).
- Карточки Истории читаются без догадок: текст статуса + один визуальный сигнал.
- Specs выровнены с реальными именами enum.

**Non-Goals:**
- Не менять enum `ReminderStatus` и не делать миграцию Room.
- Не схлопывать DONE и CANCELLED в один terminal-статус.
- Не убирать кнопку «Отменить» из уведомления (оставляем 3 действия, только яснее подписи).
- Не менять urgency-шкалу Предстоящих (NORMAL / URGENT / CRITICAL / OVERDUE) — только отделить её от terminal-статусов Истории.
- Не трогать `DismissBehavior` (свайп шторки → DONE или snooze 15).

## Decisions

### D1. Пользовательская модель статусов (канон)

```
PENDING ──fire──▶ TRIGGERED ──«Выполнено»──▶ DONE
   │                  │
   │                  └──«Отменить»──▶ CANCELLED
   │
   └──(свайп списка / детали)──▶ CANCELLED

«+10 мин» / пресеты snooze → снова PENDING (новый fireAt)
```

| Статус | Лейбл UI | Смысл для пользователя |
|--------|----------|------------------------|
| `PENDING` | Ожидает | Ещё не сработало (или отложено) |
| `TRIGGERED` | Без ответа | Сработало, пользователь не нажал действие |
| `DONE` | Выполнено | Нажал «Выполнено» |
| `CANCELLED` | Отменено | Нажал «Отменить» / свайп удаления |

**Почему «Без ответа», а не «Сработало»:** «Сработало» звучит как успех; на деле это незакрытый кейс. «Без ответа» объясняет оранжевый акцент.

**Альтернативы:** схлопнуть TRIGGERED→DONE при любом dismiss шторки — отвергнуто: теряем сигнал «проигнорировал». Переименовать enum — out of scope (без миграции).

### D2. Две независимые визуальные шкалы

```
Предстоящие (только PENDING)          История (TRIGGERED|DONE|CANCELLED)
─────────────────────────────────     ──────────────────────────────────
urgency по fireAt − now:              terminal по status:
  NORMAL   серая полоса                 DONE       зелёный + зачёркнуто
  URGENT   жёлтая                       CANCELLED  серый
  CRITICAL красная                      TRIGGERED  оранжевый
  OVERDUE  красная пульс + «просрочено»
```

Urgency **не** применяется в Истории. Terminal-цвета **не** применяются в Предстоящих.

### D3. Подписи времени/даты

| Контекст | Функция | Поведение |
|----------|---------|-----------|
| Предстоящие, `fireAt < now` | `formatRelativeDateShort` / overdue | «просрочено» |
| Предстоящие, будущее | compact date | «сегодня», «завтра», «d MMM» |
| История (любой статус) | новая `formatHistoryDate` | «сегодня» / «вчера» / «d MMM» — **без** «просрочено» |
| История, вторичная строка | `statusLabel` | «Выполнено» / «Отменено» / «Без ответа» |

`formatRelativeFireAt` в `HistoryReminderCard` больше не используется (или вызывается только для upcoming).

### D4. Кнопки действий — переименовать, не урезать

Оставляем три действия:

| Было | Стало | Статус |
|------|-------|--------|
| Готово | **Выполнено** | DONE |
| +10 мин | +10 мин (без изменений) | PENDING |
| Отменить | Отменить | CANCELLED |

**Почему не убирать «Отменить» из шторки:** для режима Уведомление шторка — основной UI; без «Отменить» пришлось бы открывать приложение. Переименование «Готово»→«Выполнено» снимает путаницу с «Отменить».

Строки: `notification_action_done`, `alarm_screen_done`, `detail_done` → «Выполнено».

### D5. Карточка Истории — явный статус

Макет:

```
│ accent │  HH:mm          body (2 lines)
│  bar   │  Выполнено · вчера
```

- DONE: зелёная полоса 3dp, body зачёркнут, muted
- CANCELLED: серая полоса 2dp, body muted, без зачёркивания
- TRIGGERED: оранжевая (TimeWarning) полоса 3dp, body обычный

Статус текстом обязателен — цвет не единственный носитель смысла.

### D6. Specs: актуальные имена статусов

В delta для `ui-screens` / `notification-delivery` / `alarm-screen-wake` заменить в затронутых requirements:
- `SCHEDULED` → `PENDING`
- `FIRED` → `TRIGGERED`
- `DISMISSED` → `DONE`
- snooze: `SNOOZED → SCHEDULED` → «статус остаётся/становится `PENDING`»

## Risks / Trade-offs

- **[Risk]** Пользователи привыкли к «Готово» → **Mitigation**: «Выполнено» однозначно; меняем везде сразу.
- **[Risk]** «Без ответа» может звучать обвиняюще → **Mitigation**: нейтральная альтернатива на ревью — «Не закрыто»; стартуем с «Без ответа».
- **[Trade-off]** Три кнопки в шторке остаются — чуть больше шума, но сохраняем полный контроль без открытия приложения.
- **[Risk]** Старые specs/docs с `SCHEDULED`/`FIRED` останутся вне delta → **Mitigation**: в этом change правим только затронутые requirements; полный audit docs — отдельный follow-up.

## Migration Plan

- Только UI/strings/specs. Данных не мигрируем.
- Rollback: откат строк и карточки Истории.
